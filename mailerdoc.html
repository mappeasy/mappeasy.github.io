Operations workflow

End-user guide

Follow this section to create and quality-check a mailing list. No knowledge of the tracing code is required.

01

Quick start

The normal sequence from input to Excel export.

Choose one input source

Select SSO GIS or Address file. The planner does not combine both sources in one run.

Prepare the locations

Filter the SSO records, upload Excel/CSV, or paste one complete address per line.

Confirm trace settings

The default is 500 ft upstream, a 70-ft parcel buffer, and one optional downstream line.

Run a small test

Use one or two locations first. Confirm that the snapped line, trace direction, and parcels make sense.

Run the full group

Watch the live progress panel until every location finishes or reports a warning.

Review and export

Inspect the map and result tables before downloading Excel.

Recommended: do not begin with 100 locations. Validate the workflow on a small sample, then run the larger group.
02

Choose the input

Two separate starting methods are available.

SSO GIS

Use records from the configured SSO ArcGIS layer.

  • The initial date window is the latest month in the dataset.
  • Filters determine which SSO points are analyzed.
  • The SSO layer switch changes map visibility only.
Address file

Use locations that are not supplied by the SSO layer.

  • Upload Excel/CSV with a column named Address.
  • Or paste one complete address per line.
  • Duplicate addresses are geocoded once and reused.
  • Review unmatched and low-score results.
Good format: 7113 EASTWOOD ST, HOUSTON, TX 77021

08

Troubleshooting

Start with the visible cause before changing service settings.

No SSO records appear

Check the date range, System, Structure Type inside Column and keyword filters, and the text filters. Remember that populated filters are combined. The SSO point layer may also be hidden.

An address does not geocode

Use a complete street address, check spelling and unit text, then review the score and matched address.

The trace starts on the wrong line

Check the input point and nearest-line radius. Look for another gravity main that is physically closer.

The trace stops

Review the warning. The normal causes are blank/broken unitid/unitid2, an endpoint gap greater than 5 ft, or no unvisited gravity-main endpoint within the recovery radius.

Too many parcels are selected

Reduce the parcel buffer, reduce upstream distance, or set downstream lines to 0.

A large run is slow

Test with the Safety limit, keep processing speed on Auto, and remember that complex branches and large parcel buffers require more GIS requests. The application limits active requests and reuses repeated queries, but service latency still matters.

T2

Tracing algorithm

ID linkage first; geometry is a validation and recovery mechanism.

1. Select the start line

findNearestGravityLine() queries ASSETGROUP = 1, measures candidates with turf.nearestPointOnLine(), and chooses the shortest point-to-line distance. Larger diameter is a tie-breaker.

2. Orient the start line

The planner looks up manhole geometry by string unitid where possible. Manhole-number lookup can help orient only the snapped line. The actual queued coordinates are the oriented line endpoints.

3. Query upstream

At current node ID N, query lines where UPPER(unitid2) IN ('N'). An accepted line's unitid becomes the next upstream node.

4. Validate endpoint

Every ID-matched candidate must have an endpoint within TRACE_ENDPOINT_TOLERANCE_FEET = 5 of current.coord.

5. Branch upstream

All unvisited, physically valid ID-supported upstream candidates may be queued. They are ordered by descending diameter, then unitid.

6. Query downstream

At downstream node ID N, query lines where UPPER(unitid) IN ('N'). The selected line's unitid2 becomes the next downstream node. Multiple downstream candidates are reduced to the largest diameter.

7. Stop controls

The snapped line counts as line 1. Distance mode can clip the last line with turf.lineSliceAlong(). visitedEdges, bestNodeDistance, and a 2,500-node guard prevent uncontrolled traversal.

Endpoint-only screening

For each returned line, the code selects the longest path, compares both endpoints to current.coord using turf.distance(), and rejects a line that merely passes near the node at midspan.

Visited-edge removal

The snapped line and accepted lines are excluded through visitedEdges. Identity priority is OBJECTID → ufid → unitid → geometry fallback.

Multiple spatial candidates

Only the closest unvisited endpoint is selected. Equal endpoint distances use larger diameter as the tie-breaker.

No candidate

The branch stops. The planner does not continue using upstreammanholenumber or downstreammanholenumber.

Closest-only applies only to spatial recovery. A normal upstream ID query may still follow several valid lines entering the same node.
T4

Performance for large SSO groups

A 100-location run can generate many dependent service calls; the planner controls both concurrency and duplication.

Fallback-only geometry query

A spatial endpoint request is not issued at every node. It runs only when the normal unitid/unitid2 path fails validation.

Location concurrency

Auto mode normally processes approximately 2–4 input locations simultaneously, depending on device and connection.

Global request limiter

MAX_CONCURRENT_GIS_REQUESTS = 8 caps all active ArcGIS requests, including nested parcel queries.

In-flight coalescing

Concurrent traces requesting the same node, manhole, nearest-line area, or five-foot endpoint wait for one shared promise instead of issuing duplicate requests.

Run-level caches

lineNodeCache, manholeNodeCache, nearbyLineQueryCache, and spatialLineCache reuse completed requests during the run.

Cache reset

Run-level caches are cleared before a new analysis and whenever service URLs change, preventing stale results from crossing configurations.

Expected effect: when topology is clean, most trace steps use cached/batched ID queries and never call the spatial fallback. Performance is therefore driven mainly by unique nodes, parcel polygons, service latency, and trace size—not simply by the number of SSO inputs.
T5

Buffer and parcel selection

The mailing candidates are geometry intersections, not confirmed recipients.

Trace geometry

The snapped start line, accepted upstream lines, and optional downstream line are combined into the result geometry.

Buffer

turf.buffer() creates the parcel-selection polygon using the configured number of feet (default 70 ft).

Parcel query

The parcel service is queried with the buffer polygon and an intersects relationship.

Mailing fields

Addresses are built primarily from Site_addr_1, Site_addr_2, and Site_addr_3; parcel identifiers and class information support QA/QC.

T7

Limitations and required judgment

The planner produces a reviewable candidate list, not an automatic final mailing list.

Five-foot recovery is conservative

It reduces false jumps to nearby parallel lines, but a real GIS connection with endpoints more than 5 ft apart will stop and create a warning.

Node IDs can still be wrong

Duplicate or incorrect unitid/unitid2 values are rejected only when geometry exposes the error. A wrong ID attached to a physically nearby line can still require map review.

Spatial proximity does not prove flow direction

The fallback chooses the closest endpoint, but geometry alone cannot fully establish upstream/downstream direction. Closest-only recovery is intentionally limited to one line.

Geocoding is not perfect

An uploaded address may match the wrong place or return a low score.

Parcel selection is geometric

A parcel is selected because its geometry intersects the buffer; outreach eligibility still requires staff review.

Data changes after export

SSO, sewer, parcel, and geocoder data may change after the workbook is created.
